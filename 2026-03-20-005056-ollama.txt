  1. Win + I 打开设置
  2. 应用 → 已安装的应用
  3. 搜索 Ollama，点击右侧 ... → 卸载

  1. Win + R 输入 appwiz.cpl 回车
  2. 找到 Ollama → 右键卸载

  # 删除模型和配置数据（默认在用户目录下）

  C:\Users\<你的用户名>\.ollama，这个文件夹里存着下载的所有模型，可能占用几十
  GB，记得清掉。

  问题诊断：KV Cache 爆了导致图片处理失败

  关键错误链：

  init_batch: failed to prepare attention ubatches
  decode: failed to find a memory slot for batch of size 512
  → failed to decode image
  → mtmd_helper_eval failed with status 1

  原因分析：

  ┌──────────────────────┬────────┬─────────────────────────────────────────┐
  │         指标         │   值   │                  说明                   │
  ├──────────────────────┼────────┼─────────────────────────────────────────┤
  │ n_ctx_slot           │ 8192   │ 你设的上下文窗口                        │
  ├──────────────────────┼────────┼─────────────────────────────────────────┤
  │ 发送图片时已用       │ ~5226  │ 对话历史已经占了大半                    │
  │ tokens               │        │                                         │
  ├──────────────────────┼────────┼─────────────────────────────────────────┤
  │ 图片需要额外 tokens  │ ~1024+ │ 2 个 batch × 512 tokens                 │
  ├──────────────────────┼────────┼─────────────────────────────────────────┤
  │ 剩余空间             │ ~2966  │ 看似够，但 KV cache 碎片化 +            │
  │                      │        │ 开销导致不够                            │
  └──────────────────────┴────────┴─────────────────────────────────────────┘

  实际上第一次纯文本请求（task 8737）就已经出现了预警 — 它不得不清掉 slot 2 的 
  653 tokens 才勉强完成。到第二次发图片时，KV cache 彻底装不下了。

  解决方案

  立即有效：

  1. 发图片前清空对话 — 新开一个会话再发图，别带着一大串聊天记录发
  2. 减少上下文长度 — 把 Context Length 从 8192 降到 4096，反而能避免 KV cache
  碎片化导致的 OOM

  3. 上下文调到 4096，对话轮数控制在 3-4 轮以内再发图
  4. 如果一定要长对话 + 图片，换 Qwen3.5-4B Q8_0，模型小，留给 KV cache 的 VRAM

  精确诊断

  Qwen3.5-9B Q4_K_M 的 VRAM 占用：

  ┌───────────────────────┬───────────┐
  │         组件          │   占用    │
  ├───────────────────────┼───────────┤
  │ 模型权重              │ ~5.6 GB   │
  ├───────────────────────┼───────────┤
  │ KV Cache (8192 ctx)   │ ~1.2 GB   │
  ├───────────────────────┼───────────┤
  │ 视觉编码器 + 临时缓冲 │ ~1-1.5 GB │
  ├───────────────────────┼───────────┤
  │ 合计峰值              │ ~8.3 GB   │
  ├───────────────────────┼───────────┤
  │ 你的 VRAM             │ 11 GB     │
  └───────────────────────┴───────────┘

  看起来有余量，但日志暴露了真相 — 当对话累积到 5200+ tokens 再发图片时，KV
  cache 内部碎片化导致分配不出连续的 512 token batch 空间。

  LM Studio 调整建议

  # 针对视觉任务优化
  Context Length:     4096    (从 8192 降下来，视觉任务不需要那么长)
  KV Cache Type:      Q8_0   (如果 LM Studio 有这个选项，能省 ~40% KV 内存)
  Batch Size:         256    (从 512 降到 256，降低单次分配压力)
  Flash Attention:    ON     (必须开，减少内存碎片)

  使用习惯

  - 发图片前开新对话 — 这是最直接有效的办法，别在聊了几十轮之后突然丢图片进去
  - 图片对话控制在 2-3 轮 — 每张图片吃 ~1024 tokens，几轮下来 KV cache 就满了
  - 纯文本对话可以保持 8192 — 昨天本小姐说的 16K-32K
  是纯文本的情况，带视觉就要砍半
